## 通用维护原则

- 以学校模板为格式基础，正式提交前必须回到 Word 模板中排版。
- 课题来源写“自拟”。
- 课题名称、项目范围、进度计划、参考文献和技术路线必须互相一致。
- 写作时围绕“基于 Flutter 与 Python 的跨平台数字笔记系统的设计与实现”，不要写成泛泛的软件开发心得。
- 正式任务书和开题报告正文不要出现“项目尚未补充”“后续再补”“保留题目合理性”等内部维护措辞。
- 需要区分实际实现与计划设计时，应使用“Python 模块设计”“接口规划”“扩展方案”“按实际完成情况纳入测试”等正式表述，不能把设计内容写成已完成系统功能。
- 若 `docs/development/ROADMAP.md`、`docs/development/POST_MVP_ROADMAP.md` 或项目实际实现范围变化，必须同步检查任务书和开题报告。
- 参考文献必须服务于“课题意义、国内外发展状况、技术可行性”论证，不能只列技术教程。
- 正式参考文献优先选用近 3-5 年成果；经典文献仅在无法被近年研究替代且正文确有需要时保留。
- 参考文献表中的每一篇文献都必须在正文中有对应引用，正文论述应与论文研究对象或结论直接相关，禁止只列不用或用一组文献笼统装饰句子。
- 正式参考文献主清单优先使用可通过学校图书馆期刊导航 `https://findshu.libsp.cn/#/magazineNav` 检索到刊名的学术期刊论文。
- 学校图书馆期刊导航核验必须从 `https://findshu.libsp.cn/#/magazineNav` 进入，按刊名检索，并同时检查刊名和馆藏电子资源覆盖年份；若引用年份不在覆盖范围内，即使刊名可检索，也应替换或移入备选资料。
- 期刊导航主要用于核验期刊来源和馆藏覆盖区间，不能替代单篇论文详情核验；正式提交前仍需按论文数据库、DOI 或出版社页面核对题名、作者、卷期和页码。
- arXiv、官方技术文档、项目文档和网页资料只能作为辅助资料，不计入“学术期刊论文”数量要求；除非学院明确允许，否则不要放入正式参考文献主清单。
- 任务书和开题报告中的项目进度必须紧扣学院中期检查、验收、论文初稿、查重、答辩和终稿截止节点。

## 任务书撰写要求

### 封面与格式

- 封面日期格式必须与模板一致，并使用当前真实时间。
- 严格参照任务书模板填写，保留模板栏目顺序。
- 学生意见栏需要黑色水笔手写并签名。
- 学生意见可参考：“接受任务，一定按进度计划按期完成毕业设计的各个环节。”

### 课题来源、意义与主要内容

- 课题来源：自拟。
- “课题意义”应写课题立题依据和研究意义，用于论证项目的社会价值和经济价值。
- 意义部分应遵循“背景资料 -> 开发必要性 -> 项目价值”的逻辑。
- 背景资料应来自参考文献、产品现状、行业现状或可靠技术资料。
- 段落之间要有逻辑过渡，避免几个案例突然堆在一起。
- 论据和论点要分清楚：“背景”是论据，用来说明开发必要性；“意义”是由论据归纳出的结论，用来概括项目价值。
- 不能把“课题意义”写成技术教程，也不能只介绍 Flutter、Python 或某个库的用法。
- “主要内容”要列出本毕业设计具体做什么，包括需求分析、总体设计、核心模块、测试和论文总结。

### 目的、要求和主要技术指标

- “目的”应是项目目的，不是作者个人学习目的。
- 目标必须具体可操作，说明系统预期达到什么效果、完成哪些功能。
- 如果写个人能力提升，只能作为补充，不能替代项目目标。
- 软件技术指标分为功能指标和非功能指标。
- 功能指标说明系统提供的功能和用途。
- 非功能指标说明性能、可靠性、安全性、扩展性、部署方便性、可用性等质量要求。
- 对本项目而言，应至少检查以下指标是否覆盖：
  - 笔记库与文件夹管理。
  - 手写画布与工具栏。
  - 多页笔记与页面管理。
  - PDF 导入、批注和导出。
  - 本地存储与恢复。
  - 文本、图片、图形、Smart Ink 等富内容能力。
  - Python 扩展模块边界。
  - 测试与验收方式。

### 进度计划

- 进度计划必须综合学校模板和学院时间表。
- 特别注意中期检查、验收和答辩时间节点。
- 计划应能对应项目阶段成果，例如需求分析、基础架构、手写核心、PDF 流程、Python 扩展规划、测试、论文和答辩材料。
- 若学院给出具体日期，应把“第 N 周”替换或补充为真实日期。

### 参考文献

- 任务书参考文献不少于 3 篇。
- 至少 1 篇英文文献。
- 参考文献要出处明确、适用性强、格式规范。
- 文献应能支持“课题意义与主要内容”的论述，不能只放技术入门教程。
- 参考文献表中的每一篇文献都应在“课题意义”或“主要内容”的论证中被实际引用。
- 正式提交版本优先选择学校图书馆期刊导航可检索到刊名的期刊论文。
- 参考文献必须编写序号。
- 文献时间尽量近，通常优先选择近 3 年左右资料；经典基础文献可以保留，但应搭配近年研究或权威技术资料。
- 可参考与课题相关的优秀硕士、博士论文，因为这类文献通常结构系统，适合作为本科毕业设计写作模仿蓝本。
- 专著、教材类参考文献不宜过多，通常最多 1 篇。
- 可以引用网页资料，但应选择权威官方网站，并写清网址和阅读日期。
- 文献尽量选择近年资料；涉及国内外研究现状时尤其要注意时效性。
- 任务书参考文献不得只列 Flutter、Python、框架 API 等技术教程；技术资料可保留，但至少应有能支撑项目意义和研究背景的文献。

## 开题报告撰写要求

### 本选题的意义

- 意义应围绕本项目主题，不宜过大或过小。
- 意义部分不必写得过长，正式提交稿应以简洁段落概括课题立题依据和研究价值。
- 需要阐述科学价值、社会价值或经济价值。
- 不要把数字笔记系统写成所有教育信息化、所有人工智能或所有办公软件的发展。
- 对本项目而言，意义应聚焦在跨平台数字笔记、手写输入、PDF 批注、本地可靠保存、搜索识别扩展和知识整理。
- 选题意义应与“基于 Flutter 与 Python 的跨平台数字笔记系统”直接相关，避免下笔过远，也避免只写某一个小功能。

### 国内外发展状况

- 先介绍国内发展状况，再介绍国外发展状况。
- 撰写时要注意角色定位：不是写产品测评，也不是写单篇论文摘要，而是围绕课题说明行业现状、用户需求和技术可行性。
- 需要围绕本项目相关内容综合概述，再进行案例论证。
- 案例出现前最好先分类，并用过渡语串联，避免突然罗列产品或文献。
- 通常应列举国内和国外分别 3 个以上案例；如果不足 3 个，需要说明原因。
- 国内案例可围绕在线文档、知识库、PDF 工具、厂商笔记生态等类别。
- 国外案例可围绕 Goodnotes、Notability、OneNote、reMarkable、Kindle Scribe、手写识别与语音笔记研究等类别。
- 国内外发展状况的目的，是论证本项目采用的技术满足用户需求、具有可操作性并有发展前景。
- 正文不需要逐篇概括参考文献，而要提取对本项目有证明意义的观点、数据、案例或结论。
- 正文中的观点和资料应尽量标注文献序号；正文可只标注文献序号，不必写作者姓名。
- 参考文献序号尽量按正文首次出现顺序排列，正文调整后应同步重排。
- 引用参考资料时要“为我所用”，重点提取能证明项目价值、技术可行性、发展方向和用户需求的材料。

### 研究内容（论文撰写提纲）

- 需要先复习软件工程相关知识，再用软件工程思维撰写。
- 提纲应参考学校毕业设计论文模板。
- 一级标题通常写作“第一章、第二章”；二级标题通常写作“1.1、1.2”。
- 开题报告中的章节名称和数量后续论文撰写时可以调整。
- 对本项目而言，提纲建议覆盖：
  - 绪论。
  - 需求分析与可行性分析。
  - 系统总体设计。
  - Flutter 前端关键模块设计与实现。
  - Python 扩展模块设计。
  - 系统测试与结果分析。
  - 总结与展望。

### 研究方法、手段及步骤

- 该部分应参考任务书中的进度计划。
- 研究方法应体现文献研究、产品比较、软件工程方法、原型迭代和测试验证。
- 技术手段应说明 Flutter、Dart、Python、本地存储、PDF 渲染、PDF 导出和 Python 扩展模块设计。
- 进度计划应与任务书一致，不能出现任务书和开题报告两个版本互相矛盾。

### 参考文献

- 开题报告参考文献不少于 10 篇学术期刊论文。
- 至少 1 篇英文原文。
- 若使用预印本、官方文档、项目文档或网页资料，应确认学校是否允许；必要时移入“技术资料”附表。
- 参考文献必须与本项目相关，服务于项目价值、技术可行性和发展方向论证。
- 参考文献表中的每一篇文献必须在正文中有可定位的对应引用，引用处应准确概括其研究对象、结论或技术方向。
- 不要写成参考文献读后感。
- 开题报告正式参考文献主清单应优先保留学校图书馆期刊导航可检索到刊名的学术期刊论文。

## 参考文献格式要求

### 期刊文章

```text
[序号]作者姓名. 文题[J]. 刊名, 年, 卷(期): 起讫页.
```

作者不超过 3 人时全部写出；超过 3 人时写前 3 名，后加“等”或“et al”。

### 图书

```text
[序号]作者姓名. 书名[M]. 版次(第 1 版略). 出版地: 出版者, 年. 起讫页.
```

### 学位论文、研究报告、标准等

可按类型标识填写：

- 专著：`M`
- 论文集：`C`
- 报纸文章：`N`
- 期刊文章：`J`
- 学位论文：`D`
- 研究报告：`R`
- 标准：`S`
- 专利：`P`
- 其他：`Z`

### 网上资料

```text
[序号]作者或机构. 文章名或页面名[EB/OL]. 网址, 阅读日期.
```

网页资料应优先使用官方文档、权威机构或可信项目页面。

## 更新前检查清单

每次修改任务书或开题报告前，先确认：

- 课题名称是否仍为“基于 Flutter 与 Python 的跨平台数字笔记系统的设计与实现”。
- Flutter 客户端已实现内容和 Python 模块设计/实现状态是否区分清楚。
- 任务书和开题报告中的进度计划是否一致。
- 技术指标是否分成功能指标和非功能指标。
- 开题报告是否先写国内，再写国外。
- 国内外发展状况是否有分类和案例支撑。
- 参考文献数量、英文文献、格式和时效性是否满足要求。
- 正式参考文献是否优先采用近 3-5 年成果，经典文献是否确有不可替代的正文用途。
- 每一条参考文献是否已在正文中实际引用，引用语义是否与文献内容对应。
- 参考文献主清单是否已通过学校图书馆期刊导航核验期刊来源和引用年份对应的馆藏覆盖范围。
- 学院中期检查、验收、答辩时间是否需要更新。
- `docs/development/STATUS.md` 是否需要记录本次学业文档同步。

## 更新后检查清单

每次修改后至少检查：

- `git diff --check`
- `rg -n '[ \t]$' docs/academic docs/development/STATUS.md`
- 路径引用是否仍指向 `docs/development/`、`docs/academic/` 或 `docs/learning/` 下的新目录。

如果修改内容影响毕业设计范围，应同步更新：

- `docs/academic/GRADUATION_TASK_BOOK.md`
- `docs/academic/OPENING_REPORT.md`
- `docs/development/STATUS.md`
- 必要时更新 `docs/development/ROADMAP.md` 或 `docs/development/POST_MVP_ROADMAP.md`
